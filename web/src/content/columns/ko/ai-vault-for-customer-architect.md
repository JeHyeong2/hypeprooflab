## 수요일 저녁, 3시간짜리 회의가 끝났다

현대차 SDV 팀, 42dot, 내부 엔지니어링 — 세 방향에서 쏟아진 정보가 머릿속에서 뒤엉켜 있었다. 42dot의 XP2 양산이 심각한 위기라는 것, 주형진 상무가 그룹 차원에서 파견됐다는 것, Applied Intuition의 PoC가 "다 별로"라는 평가를 받았다는 것, Ford 레퍼런스가 HKMC 피칭에 쓸 수 있다는 것 —

이 모든 걸 어디에 적어야 할까?

예전의 나라면 노션에 회의록을 복붙하고, 슬랙에 "중요: XP2 이슈" 메시지를 보내고, 그리고… 2주 뒤에 "그때 42dot 상황이 어땠더라?" 하고 슬랙을 뒤졌을 것이다. 검색해도 안 나오고, 노션 페이지는 이미 10개가 넘어가서 어디에 뭘 적었는지 모른다.

지금은 다르다. 회의록을 하나의 파일로 던지면, AI가 14개의 시그널을 자동 추출하고, 경쟁사 프로필을 업데이트하고, 내일 아침 브리핑에 반영한다. 내가 만든 건 **Knowledge Vault** — AI 에이전트들이 관리하는 업무 기억 시스템이다.

---

## 왜 기존 도구로는 안 되는가

Customer Architect라는 역할은 좀 특수하다. 고객사(현대차), 경쟁사(42dot, Applied Intuition), 내부 제품(CCU 2.0, AI Director 등 8개), 이해관계자(20명+) — 이 네 축의 정보가 매일 뒤섞여서 들어온다. 하나의 회의에서 동시에:

- 경쟁사 동향 ("Applied Intuition이 6주 PoC 제안했대")
- 고객 pain point ("42dot DDS 통신 완전히 죽었음")
- 내부 제품 maturity 갭 ("우리 D3인데 고객은 D4 기대")
- 액션 아이템 ("Chris가 2차 미팅 잡아야 함")
- 이해관계자 역학 ("이동렬 상무 — 숫자 없으면 진행 불가")

이게 전부 하나의 대화에서 나온다. 노션으로? 어느 데이터베이스에 넣을지부터 고민이다. Jira로? 이건 티켓이 아니라 인텔리전스다. CRM으로? 이해관계자 역학까지 추적하는 CRM은 없다.

결국 필요한 건 **구조화된 기억 시스템**이었다. 그것도 내가 일일이 분류하지 않아도 되는.

## Vault의 구조 — 세 개의 방

Vault는 세 개의 영역으로 나뉜다. 비유하자면 **서재, 도서관, 일기장**이다.

### 🎯 서재 (jay/) — 내 업무 공간

```
jay/
├── agendas/          ← 제품별 Status Card (8개 제품 + 11개 Use Case)
├── signals/          ← 월별 시그널 로그 (append-only)
├── stakeholders/     ← 이해관계자 맵
└── reports/          ← AI가 생성한 브리핑
```

서재에는 **내가 지금 추적하는 것들**이 있다. 제품마다 Status Card가 있다 — CCU 2.0의 현재 maturity는 D3이고 고객은 D4를 기대한다, 블로킹 이슈는 뭐고, 데모 계획은 어떻고. 이걸 매일 AI가 업데이트한다.

시그널 로그는 한 달치를 하나의 파일에 쌓는다. append-only라서 절대 덮어쓰지 않는다. 2월에 42dot에서 어떤 신호가 나왔는지 한눈에 보인다.

### 🧠 도서관 (ca-knowledge/) — 축적형 지식

```
ca-knowledge/
├── products/         ← 제품 지식 (CCU2, AI Director 등)
├── customers/        ← 고객 프로필 (HKMC)
├── competitors/      ← 경쟁사 (42dot, Applied Intuition)
└── ontology-index.yaml  ← 자동 생성 검색 인덱스
```

도서관은 시간이 지나도 유효한 지식이 쌓이는 곳이다. 42dot의 프로필에는 "송창현 사장 퇴임", "PM 인력 이탈", "XP2 양산 위기" 같은 정보가 날짜와 함께 축적된다. 6개월 뒤에 42dot 전략을 분석할 때, 이 파일 하나면 된다.

`ontology-index.yaml`은 도서관의 카탈로그다. AI가 자동으로 생성하는 검색 인덱스로, "VCRM 소유부서가 어디야?"라고 물으면 관련 파일을 찾아서 답해준다.

### 📝 일기장 (daily-retrospective/) — 매일의 기록

```
daily-retrospective/
└── 2026-02-26.md     ← 오늘 뭘 했고, 뭘 놓쳤는지
```

하루가 끝나면 AI가 자동으로 회고를 작성한다. 오늘 어떤 시그널이 들어왔고, 어떤 액션이 밀렸고, 시스템에 어떤 문제가 있었는지. 사람이 쓰는 일기가 아니라, AI가 쓰는 업무 일지다.

## 마법의 순간 — 회의록 하나로 14개 시그널이 뽑힌다

수요일 저녁의 그 회의로 돌아가자. 나는 회의록을 `data/meeting-notes/`에 던졌다. 그리고 `/ca-ingest`를 실행했다.

뒤에서 일어나는 일:

**Phase 1 — 4개 에이전트가 동시에 읽는다**

| 에이전트 | 하는 일 | 결과 |
|---------|--------|------|
| Signal Extractor | 10가지 타입의 시그널 추출 | 14개 시그널 발견 |
| Maturity Assessor | 제품별 D0-D5 성숙도 평가 | CCU 2.0: D3 (고객 기대 D4) |
| Action Extractor | 액션 아이템 + 우선순위 + 담당자 | P1 2개, P2 4개, P3 1개 |
| Stakeholder Mapper | 이해관계자 역학 변화 감지 | 주형진 상무 역할 업데이트 |

4개 에이전트가 같은 문서를 동시에 읽지만, 각자 다른 렌즈로 본다. Signal Extractor는 "DDS 통신 완전히 죽었음"에서 `pain_point`를 감지하고, Stakeholder Mapper는 "주형진 상무가 그룹 차원에서 파견됐다"에서 조직 역학 변화를 읽는다.

핵심은 에이전트들이 **전부 read-only**라는 것이다. 파일을 읽고 분석만 한다. 아무것도 쓰지 않는다.

**Phase 2 — 기존 지식과 대조한다**

| 에이전트 | 하는 일 |
|---------|--------|
| Agenda Mapper | Phase 1 결과를 해당 제품 Status Card에 매핑 |
| Vault Differ | "이건 새 정보야? 업데이트야? 이미 알고 있는 거야?" 판별 |

Vault Differ가 가장 중요하다. "42dot DDS 문제"가 들어오면, 기존 `ca-knowledge/competitors/42dot.md`를 읽고 비교한다. "DDS 리소스 문제는 이미 있지만, 차량 시동 불가까지 갔다는 건 새 정보"라고 판단하면 `update`로 분류한다.

**Phase 3 — 스킬이 Vault에 쓴다**

에이전트들이 분류를 마치면, `/ca-ingest` 스킬이 결과를 받아서:

- `jay/signals/2026-02.md`에 14개 시그널 append
- `jay/agendas/ccu2.md` Status Card 업데이트
- `ca-knowledge/competitors/42dot.md`에 새 이슈 추가
- `jay/stakeholders/stakeholder-map.md` 역할 변경 반영

그리고 자동으로 git commit. 모든 변경이 이력으로 남는다.

## 아침 7분 — 모닝 브리핑

다음 날 아침 9시. 크론잡이 돌아간다.

```
OpenClaw Cron → Walter(에이전트) → run-ca.sh morning → Claude Code headless → 브리핑 생성
```

AI가 만든 모닝 브리핑이 Discord DM으로 온다:

> **즉시 주의 (P0/P1)**
> - 주형진 상무 2차 미팅 준비 — VCRM/SDV 아키텍처 (Chris 조율 중)
> - 4월 초 Korea 방문 + CCU팀 워크샵 준비 시작
>
> **핵심 시그널 (2/25-2/26)**
> - XP2 위기: 다인산업 수조립, DDS 통신 실패, 중국 배터리 SOC 문제
> - 42dot 인력 공백: PM/구매 이탈, 커뮤니케이션 채널 불명확
> - VCRM 기회: 소유 부서 부재 → greenfield opportunity

7분이면 충분하다. 어제 3시간 회의에서 내가 집중해야 할 것, 이번 주 안에 해야 할 것, 경쟁사 상황 변화가 한 장에 정리되어 있다. 예전에는 이걸 만들려면 노션 5개 페이지를 뒤져야 했다.

## 10가지 시그널 — 업무의 약한 신호를 잡는다

이 시스템의 핵심은 **시그널 분류 체계**다. 회의에서 오가는 말들 중 90%는 잡담이고, 10%가 의미 있는 신호다. AI는 10가지 타입으로 분류한다:

| 시그널 | 의미 | 예시 |
|--------|------|------|
| `feature_request` | 고객이 원하는 것 | "UDS 기반 통합 진단 공동개발 제안" |
| `pain_point` | 고객이 아파하는 것 | "XP2 DDS 통신 완전히 죽었음" |
| `timeline_pressure` | 시간 압박 | "Jeff 4월 초 Korea 방문" |
| `competitor_mention` | 경쟁사 언급 | "Applied Intuition 6주 PoC, 가격 할인" |
| `architecture_gap` | 기술 격차 | "VCRM 방향성조차 부재" |
| `satisfaction` | 만족 신호 | "Ford Data Collector 계약 체결" |
| `escalation` | 에스컬레이션 | "주형진 상무 그룹 차원 파견" |
| `strategic_hint` | 전략적 암시 | "XP2는 양산이 아니라 R&D 차량" |
| `budget_signal` | 예산 신호 | "ROI 수치 없으면 진행 불가" |
| `maturity_expectation` | 성숙도 기대 | "즉시 적용 가능한 솔루션 요구" |

하나의 회의에서 14개 시그널이 나왔다. 이 중 `pain_point`가 4개, `strategic_hint`가 2개, `escalation`이 1개. 이 분포만 봐도 "42dot이 지금 심각한 상황이고, 이게 우리에게 기회"라는 패턴이 보인다.

한 달치 시그널을 모아서 트렌드를 보면 더 강력하다. "2월 초에는 `feature_request`가 주류였는데, 중순부터 `pain_point`와 `escalation`이 급증" — 이런 패턴은 사람이 수동으로 추적하면 절대 안 보인다.

## Maturity Framework — D0부터 D5까지

제품 성숙도를 6단계로 정의했다:

| 레벨 | 이름 | 한마디 |
|------|------|--------|
| D0 | Concept | 슬라이드만 있음 |
| D1 | Recorded Demo | 녹화된 데모 |
| D2 | Interactive Prototype | 실시간 데모 가능 |
| D3 | Integration-Ready | 고객 환경 연동 설계 완료 |
| D4 | Pilot-Ready | 고객 환경에서 파일럿 가능 |
| D5 | Production-Ready | 양산/운영 준비 완료 |

CCU 2.0은 내부적으로 D3이다. 컨테이너 매니저가 돌아가고, 여러 하드웨어 플랫폼을 지원한다. 하지만 고객(현대차)은 D4를 기대한다. "즉시 적용 가능"해야 한다는 것이다.

이 갭을 Status Card에 명시적으로 적어두면, 모닝 브리핑에서 "CCU 2.0: D3→D4 갭 해소 필요"라고 매일 리마인드해준다. 잊을 수가 없다.

## 자가 개선 — AI가 AI를 평가한다

일주일에 한 번, 일요일 새벽에 조용히 돌아가는 파이프라인이 있다:

```
일요일 02:00  /review    → 시스템 자가 비평 (7개 차원, 점수화)
일요일 03:00  /remediate → critical 항목 자동 수정
일요일 03:30  /ca-qa-harness → Q&A 품질 벤치마크
```

`/review`는 critic 에이전트가 전체 시스템을 평가한다. "데이터 품질 B-, 에이전트 정확도 C+, 지식 관리 B" 같은 식으로. Critical이나 High 이슈가 발견되면 `/remediate`가 자동으로 고친다.

`/ca-qa-harness`는 더 재밌다. "42dot의 현재 상황은?" 같은 질문을 던져서, AI가 vault에서 정확한 답을 찾는지 테스트한다. 사실 정확성, 소스 인용, 관련성을 점수화하고, 점수가 떨어지면 온톨로지 인덱스를 재구축한다.

AI가 자기 자신의 기억력을 시험보는 셈이다.

## 실패담 — 이 시스템이 완벽하지 않은 이유

솔직하게 말하자. 이 시스템은 아직 깨지기 쉽다.

**크론이 죽는다.** CA Morning 크론이 이틀째 실패 중이다. Claude Code headless 세션 생성이 안 되면 아침 브리핑이 안 온다. 아침에 일어나서 "오늘 브리핑이 왜 안 왔지?" 하고 직접 확인해야 한다.

**시그널 추출이 100%가 아니다.** 한국어-영어 혼용 회의록에서 맥락을 놓치는 경우가 있다. "별로였다"가 부정적 평가라는 걸 AI가 항상 잡지는 못한다.

**Vault가 무한히 커진다.** 매일 시그널이 쌓이고, 제품 카드가 업데이트되고, 리포트가 생긴다. 3개월만 지나면 파일이 수백 개다. 아직 아카이빙 정책이 없다.

**혼자 쓸 때만 잘 된다.** 이 시스템은 철저히 1인용이다. 팀원이 같이 쓰려면 conflict resolution, 권한 관리, 동시 편집 — 노션이 이미 해결한 문제들을 다시 풀어야 한다.

---

## 그래서 왜 이걸 쓰는가

노션이 있고, Confluence가 있고, CRM이 있는데 왜 마크다운 파일과 AI 에이전트로 이런 짓을 하는가?

**첫째, 자동 분류.** 회의록을 던지면 끝이다. 어느 데이터베이스에 넣을지 고민하지 않아도 된다. AI가 시그널을 뽑고, 해당 제품 카드에 매핑하고, 경쟁사 프로필을 업데이트한다.

**둘째, 맥락이 살아있다.** 42dot의 DDS 문제가 처음 나왔을 때는 "리소스 문제"였다. 한 달 뒤에는 "차량 시동 불가"로 악화됐다. Vault Differ가 이걸 `update`로 분류하면서 히스토리가 자연스럽게 쌓인다.

**셋째, 아침 7분.** 모닝 브리핑 하나로 어제의 모든 변화를 파악한다. 이게 없으면 슬랙, 이메일, 노션을 각각 열어서 30분을 써야 한다.

**넷째, git으로 관리된다.** 모든 변경이 커밋 히스토리에 남는다. "2주 전에 이동렬 상무에 대한 평가가 어떻게 바뀌었지?"를 `git log`로 추적할 수 있다. 노션에서는 불가능한 일이다.

## 당신의 Vault는 어떤 모양일까

이 시스템은 Customer Architect라는 특수한 역할에 맞춰졌다. 하지만 **구조화된 기억이 필요한 모든 역할**에 응용할 수 있다.

- **영업**: 고객 시그널 추적 + 딜 파이프라인 maturity
- **PM**: 제품 피드백 분류 + 스프린트 리스크 감지
- **리서치**: 논문/뉴스에서 트렌드 시그널 추출 + 주간 브리핑
- **컨설턴트**: 프로젝트별 이해관계자 맵 + 이슈 추적

중요한 건 도구가 아니라 **구조**다. 어떤 시그널을 추적할 것인가, 지식을 어떤 축으로 분류할 것인가, 어떤 주기로 리뷰할 것인가. 이 세 가지가 정해지면 마크다운 파일이든 노션이든 Obsidian이든 — 어디에 구현하든 작동한다.

다만, AI 에이전트가 자동으로 분류하고 브리핑을 만들어주는 맛을 한번 보면… 수동으로 돌아가기는 어렵다.
